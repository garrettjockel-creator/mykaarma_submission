Architecture track  ·  Stage: Discover

Architecture / platform one-pager

For platform, infrastructure, and architectural constraint work. The "customer" here is the product roadmap and the engineering team. Technical opinion and measured constraint evidence are not the same thing.

i

Submission details

Must be a named individual — not "the team".

1

Constraint statement

Pass / fail
State the specific technical constraint in measurable terms. What cannot be built, scaled, or maintained because this limitation exists? Name the system, the failure mode, and at what threshold it occurs.

Example: "The current video storage metadata schema uses a flat table structure that cannot support per-dimension scoring queries across 2,000+ dealers without full table scans. At current TVG submission volume (~14,000 videos/day), P95 query latency for the scoring dashboard exceeds 18 seconds — 9× our 2-second UI response target."
⚠"Our architecture is getting hard to maintain" is an opinion. "P95 latency degrades to 12 minutes at 1,800 concurrent dealers, breaching our 8-minute SLA" is a constraint. Only the second advances.
0 words
Checking constraint quality...
Pass / fail: Must name a specific system. Must state a measurable failure mode or ceiling. Must identify the threshold at which the constraint becomes real. "Technical debt" with no specifics fails.
2

Evidence quality — technical

Max 4 pts
Select the highest tier you can substantiate with evidence you currently hold.
Tier 1 — Opinions
Engineer self-assessment with no supporting data. "This feels fragile." "We will need to fix this eventually." Architectural intuition without measurement.
0 pts
Tier 2 — Assessment
Team or lead engineer review. Architecture diagram analysis. Code review findings. External tech review. Feasibility evaluation.
1 pt
Tier 3 — Observed symptoms
Support escalations attributable to the constraint. Repeated workarounds by engineers. Features deferred because of the limitation. Incidents linked to the system.
2 pts
Tier 4 — Measured constraint
Benchmark data showing actual degradation. Load test results. Profiling output. SLA breach evidence. Specific query latency, memory usage, or error rate data at scale.
3 pts
Tier 5 — Validated at scale
Production data showing constraint under real load. Multiple engineers independently confirming the measurement. Incident post-mortems directly attributable to the limitation.
4 pts
Tier 1 auto-fails. This submission will not advance regardless of total score.
Rule: Tier 1 scores 0 and does not advance. Tier 2 may advance for low-cost investigations only. Tier 3+ required for any significant engineering investment.

2b — Specific technical evidence (required for Tier 2 and above)

2c — Engineering signal (required)

No named engineer who has assessed the constraint + no identified blast radius = submission does not advance.

3

Root cause hypothesis

Max 2 pts
Why does this constraint exist? Explain the architectural or design decision that created it — and why that decision was reasonable at the time. Then explain what has changed that makes it a problem now.

Example: "The flat metadata schema was designed when TVG processed ~200 videos/day at launch. The assumption was that query patterns would remain simple. The introduction of per-dimension scoring requires joins across 5 tables at 14,000 videos/day — a load profile the schema was never designed to handle."
Checking hypothesis quality...
Pass / fail: Must explain why the constraint exists and what changed. "Legacy code" with no further explanation fails.
4

Product bets blocked or at risk

Max 2 pts
Requirement: Architecture work without a named product bet it unblocks is overhead, not strategy. Name the specific funded initiatives that cannot proceed, cannot scale, or are at risk because of this constraint.
Pass / fail: Must name a specific funded initiative. "Future flexibility" or "general platform health" scores 0 — these are not product bets.
5

Investigation ask

Max 1 pt
Two to three questions the investigation will answer about the constraint — not the solution. The investigation produces an A3 and a six-pager recommendation, not a solution design.

Example questions: What is the full scope of schema changes required to support per-dimension scoring at 2,000 dealer scale? What is the migration risk to existing TVG video delivery if the schema is refactored? What are the build-vs-refactor options and what does each cost in engineering time?
Hard stop: No solution design. No technology selection. No engineering estimate. Questions that presuppose a solution ("How do we migrate to Postgres?") are out of stage — rephrase as constraint questions ("What are the options for resolving the query latency constraint?").
